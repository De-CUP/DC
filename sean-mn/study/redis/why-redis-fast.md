
## 왜 레디스는 그렇게 빠른가?

레디스의 압도적인 속도는 단순히 In-Memory 저장소뿐만 아니라 우아한 저수준 설계 덕분이다.

### 싱글 스레드 기반 이벤트 루프
레디스는 한 번에 하나의 명령만 처리하는 단일 스레드 모델을 사용한다. 기존의 select/poll 방식은 모든 소켓을 순차적으로 순회하므로 연결 수가 늘어날수록 성능이 선형적으로 저하된다. 반면 epoll은 커널이 이벤트가 발생한 소켓만 골라서 알려주므로 수만 개의 클라이언트 연결도 O(1)에 가까운 비용으로 처리할 수 있다. 이를 통해 Lock이나 컨텍스트 스위칭 오버헤드 없이 고성능 I/O를 달성한다.

### C언어와 정밀한 메모리 관리
C언어로 작성된 레디스는 JVM이나 Python 인터프리터 같은 런타임이 없어 언어 계층의 오버헤드가 없다. 메모리 관리는 jemalloc을 사용하는데, jemalloc은 arena 기반 구조로 스레드별 메모리 풀을 독립적으로 관리하여 파편화를 억제하고 CPU 캐시 효율을 극대화한다.

### 결정론적 성능
C언어로 메모리를 수동 관리하기 때문에 가비지 컬렉터(GC) 자체가 존재하지 않는다. Java나 Go에서는 GC가 실행되는 순간 애플리케이션 스레드가 일시 정지하는 Stop-The-World(STW) 현상이 발생해 응답 시간이 수십 ms 단위로 튀는 경우가 있다. 레디스는 이 문제가 구조적으로 없어 매우 예측 가능한 저지연 성능을 보장한다.

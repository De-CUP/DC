# TL;DR

전공 이론 공부와 CS 공부만 하다보니 지쳤다. 마침 Three.js의 셰이딩 기반 언어가 GLSL이라고 하니 GLSL을 기반으로 셰이더을 알려주는 [The Book of Shaders](https://thebookofshaders.com/00/?lan=kr)를 읽고 셰이더에 대한 기초적인 개념을 익혔다.

## 셰이더(Shader)

컴퓨터 그래픽에서 3차원 장면에 빛과 어둠 그리고 색을 계산하는 프로그램.

컴퓨터 그래픽으로 무언가를 표현해야 할 때는 원, 선, 삼각형, 사각형 등을 조합하여 새로운 모양을 만들어낸다. 셰이더에서도 이 규칙은 스크린의 모든 낱개의 픽셀에 적용이 된다. 그것도 모두 한번에.

### 왜 셰이더는 빠를까?

CPU는 빠른 프로세서 몇 개로 순차 처리하지만,
GPU는 느린 프로세서 수천 개가 물리적으로 병렬 처리한다.
픽셀 연산처럼 단순하지만 양이 많은 작업에서 GPU가 압도적으로 유리한 이유다.

### 프래그먼트 셰이더(Fragment Shader)

각 픽셀(프래그먼트)의 색을 계산하는 셰이더.

## GLSL(openGL Shading Language)

대중적인 셰이딩 언어.

- C와 형태가 유사하다.
- #을 통해 전처리를 할 수 있다.
- precision으로 실수의 정확도를 표현할 수 있다.(낮을수록 속도가 빠르고, 높을수록 퀄리티가 높다.)
- 타입캐스팅이 없다. (실수를 인자로 원하는 함수에 정수를 넣으면 오류가 난다.)

## 유니폼(Uniforms)

CPU에서 GPU로 넘겨주는 읽기 전용 입력값.

- 모든 쓰레드에서 동일한 값을 공유한다.
- 실수, 벡터 등의 값을 지원한다.
- 쉐이더 연산이 시작된후부터의 초, 쉐이더가 그려지고 있는 화면의 크기, 마우스의 현재 위치값등을 가지고 있다.

## 현재 처리 중인 픽셀의 좌표(gl_FragCoord)

각 스레드(픽셀)가 화면의 어느 위치를 처리하는지 알려주는 내장 입력값.

- uniform과 달리 픽셀마다 값이 다른 varying 타입이다.
- u_resolution으로 나눠 0~1로 정규화하면 색상값으로 바로 사용할 수 있다.

## Shaping Functions

0~1로 정규화된 x 좌표를 수학 함수로 변환해 원하는 곡선을 만드는 기법.

### 선형 보간

x와 y가 1:1로 대응되는 가장 기본적인 형태.
`float y = st.x` → x가 0이면 검정, 1이면 흰색인 그라데이션.

### pow(x, n)

x를 n제곱한 곡선. n이 클수록 곡선이 급격히 휘고,
n이 1이면 직선, 0.5면 반대 방향 곡선이 된다.

### step(edge, x)

edge 기준으로 딱 잘라서 0 또는 1만 반환.
보간 없이 경계가 날카롭다.

### smoothstep(a, b, x)

a~b 구간에서 0→1로 부드럽게 보간.
범위 밖은 0 또는 1로 클램핑된다.

### sin / cos

-1~1 사이를 주기적으로 반복하는 함수.

- x에 큰 수를 곱할수록 주기가 좁아짐
- u_time을 더하면 그래프가 이동하는 애니메이션
- u_time을 곱하면 시간이 지날수록 주기가 촘촘해져 노이즈처럼 변함
- abs() 적용 시 공이 튕기는 효과
- fract() 적용 시 톱니 모양으로 반복

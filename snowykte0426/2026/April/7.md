# 2026/04/07
### Kotlin `expect` / `actual`
- 한 문장으로 정의하면 <b>"공통 코드에서 인터페이스만 선언하고, 각 플랫폼에서 구현체를 제공하는 패턴"</b>이다.
- commonMain에서는 플랫폼 API를 직접 쓸 수 없기 때문에 플랫폼마다 날짜, UUID, 파일, 로깅 API가 전부 다르다.
```kotlin
// commonMain에서 이렇게 쓰면 컴파일 에러
val now = java.time.LocalDateTime.now() // Java API — iOS에 없다
val uuid = java.util.UUID.randomUUID()
```
- `expect` / `actual`은 이 간극을 메우는 공식 메커니즘
```
src/
├── commonMain/       expect 선언 — "이런 게 있을 것이다"
├── androidMain/      actual 구현 — Android 버전
├── iosMain/          actual 구현 — iOS 버전
└── jvmMain/          actual 구현 — JVM/서버 버전
```
- #### 예시 1. UUID 생성기
```kt
// commonMain/Platform.kt
expect fun randomUUID(): String
```
```kt
// androidMain/Platform.android.kt
actual fun randomUUID(): String = java.util.UUID.randomUUID().toString()
```
```kt
// iosMain/Platform.ios.kt
import platform.Foundation.NSUUID

actual fun randomUUID(): String = NSUUID().UUIDString()
```
```kt
// jvmMain/Platform.jvm.kt
actual fun randomUUID(): String = java.util.UUID.randomUUID().toString()
```
- `commonMain` 어디서든 `randomUUID()`를 그냥 호출하면 작동한다.
- expect / actual vs 인터페이스

| 상황 | 권장 방식 |
|------|----------|
| 플랫폼 API를 직접 래핑해야 할 때 |`expect / actual` |
| 구현체가 런타임에 교체될 수 있을 때 (DI) | `interface` |
| 테스트에서 Mock이 필요할 때 | `interface` |
| 단순 유틸 함수 수준일 때 | `expect fun` |
